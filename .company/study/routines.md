# 定期実行(Routine)の指示書

> ## 🛑 全Routine一時停止中(2026-07-14 オーナー指示)
> **理由**: トークン消費が想定を大きく超えたため。
> **Routineとして起動した場合**: 以下のセクションを実行せず、Web検索もファイル作成もコミットも行わず、**そのまま終了すること**。「停止中のため何もしませんでした」の一言だけでよい。
> **再開の条件**: オーナーがこの停止ブロックを削除したとき。再開時は頻度を落とす(例: R1を週1に、R2〜R5を月1か手動起動のみに)ことを先に検討する。

学習体制の2層構造に基づく定期実行の定義。各Routineはこのファイルの該当セクションを実行指示として使う。
時刻はすべて日本時間(JST)。cron設定はUTC(JST-9時間)。

## 共通ルール(全Routine)
- 共通憲法に従う(未検証の数字を断定しない、出典を付ける)
- 成果物は保存して git commit & push(スマホから読めるように)
- 新しい発見がない日は無理に埋めず「特記事項なし」と短く記録(量産よりも採用率)
- エスカレーション条件(CLAUDE.md)に該当したら自動処理せずオーナーに報告

## R1: 全社AIニュースブリーフ — 毎朝7:00 JST (cron: 0 22 * * *)
担当: ジーナ本体(全社学習)
1. Web検索で過去24〜48時間のAI・副業関連ニュースを収集
2. 5本選ぶ(意外性・実務への影響・うちの事業への関係を優先)
3. 各本: 見出し/2文サマリー/うちへの関係(蔵書と接続)/出典リンク
4. `study/briefs/YYYY-MM-DD.md` に保存
成功条件: 5本・全て出典付き・3分で読める。蔵書級の発見は library への追加も検討。

## 部門週次調査の共通手順(R2〜R5)
1. 担当部門のCLAUDE.mdの**「学習」セクション(収集基準・集めないもの)を読む**
2. 既存の `knowledge.md` を読み、**既に知っていることの再収集を避ける**(差分だけ集める)
3. Web検索で調査し、`<部門>/research/YYYY-MM-DD.md` に記録(要点+出典+うちへの適用案)
4. 使える知見だけを `<部門>/knowledge.md` に蒸留して追記(タイムスタンプ付き)
5. 部門を超えて価値がある発見は `study/library/` への昇格を検討し、`study/index.md` を更新
6. 新しい発見がない週は「特記事項なし」と短く記録して終える
