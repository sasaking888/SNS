# 「MarkdownよりHTML」 — エージェントの成果物フォーマット論

## 出典
Claude Code開発チームのThariq氏によるClaude Blog掲載記事。今日の蔵書の中で最も出所が確かな一次情報(実例集: thariqs.github.io/html-effectiveness)。販促なし。

## 要点
- Markdownはエージェントとの標準連絡形式になったが、**100行を超えるMarkdownは人間が読まない**。エージェントが強くなるほど、仕様書・計画・レポートが長くなり、Markdownが制約になる
- 筆者はHTML出力に移行。理由: ①**情報密度**(表・CSS・SVG図・コード・インタラクション・空間配置まで表現できる。Markdownだと ASCII図やUnicode文字で色を推定するような非効率が起きる) ②**読みやすさ**(タブ・図・リンクで視覚的に構造化、モバイル対応も可) ③**共有しやすさ**(ブラウザで開けばいいので読んでもらえる確率が段違い) ④**双方向**(スライダーやコピー用ボタンを付けて、調整結果をプロンプトに戻せる) ⑤Claude Codeはファイル・MCP・git履歴など文脈を大量に取り込んで作れる ⑥楽しい=関与感が上がる

## 使い所(記事のユースケース)
1. **仕様・計画・探索**: 案を6方向並べて比較するグリッド、モックアップ+データフロー+コード付き実装計画
2. **コードレビュー・PR理解**: 差分+注釈+重要度の色分け(GitHubのdiff表示より分かりやすいことも)
3. **デザイン・プロトタイプ**: スライダー付きでアニメや色を調整→「パラメータをコピー」ボタンで結果をAIに戻す
4. **レポート・調査・学習**: Slack・コード・git履歴を横断した読みやすい報告書、SVG図解
5. **使い捨ての編集UI**: チケットのドラッグ優先順位付け、設定ファイルのフォーム編集、プロンプトのライブ調整——最後に必ず「コピー」ボタンで結果をテキスト化してAIに戻す

## 筆者自身が認める欠点(誠実)
- トークン消費が多く、生成に2〜4倍時間がかかる
- **バージョン管理に弱い**(HTMLのdiffはノイズが多くレビューしづらい)
- 筆者は「HTML極大派」と自認しており、全員に同じバランスを勧めているわけではない

## 批判的分析
- 主張は実体験ベースで、欠点も自分から開示しており信頼できる。ただし「共有しやすい」はS3等に置ける環境の話で、**GitHubモバイルで読む運用(うち)ではMarkdownが素のまま綺麗に表示される**ため、共有面の優位はうちには部分的にしか当てはまらない
- 「/htmlスキルにするな、まず素で頼め」という注意も含めて、道具より使い方の理解を優先する姿勢が一貫している

## うちの現状との照合
- **プロンプト棚のArtifact化で既に半分実践済み**(ワンタップコピー付きHTML棚)。この記事はその方向の妥当性を、Claude Codeチーム本人の実践で裏付けた形
- うちの知識ファイル(蔵書・knowledge.md・ブリーフ)はGit管理+スマホGitHub閲覧が前提なので、Markdownの「差分が読める・そのまま表示される」利点が効いている。**全面HTML化は合わない**
- 効きそうなのは「読む体験・比較判断・調整」が主目的の成果物: 運用レビューのダッシュボード、投稿案の比較選択UI、蔵書の視覚的な索引など

## 吸収の結果(2026-07-11 オーナー判断)
1. 使い分け基準のルール化: 記録・知識=Markdown / 閲覧・判断・調整用の成果物=HTML Artifact → **採用**(.company/CLAUDE.mdに反映)
2. 定期の運用レビューをHTMLダッシュボードで出す → **採用**(routines.mdに反映)
3. 「比較・選択」場面での使い捨てUI → **保留**。実際の場面で都度試し、有効と分かってからルール化(構築順序: 手動で確実に→それから固定)
