## AIを「自己解剖ツール」として使う — 直感カルテの手法

**出典**: 寺田正信氏（@terada_masanobu）のX記事。yutori社長・片石氏（ゆとりくん）のAI活用法の解説と実践レポート。

### 背景

- yutori: 創業5年8ヶ月でアパレル史上最年少上場（30歳）、年商142億超、前期比71.4%増で過去最高益
- その社長がAIを使っていたのは経営分析でも業務効率化でもなく、自分の内面を掘る「自己解剖」（直感カルテ）だった
- 紹介者の寺田氏は2016年からAmazon・楽天・Yahoo!・TEMUで自社ブランドを運営する現役EC事業者

### 核心のアイデア

1. **人間は「すでに知っている言葉・気づいている範囲」でしか思考できない**。だから言葉で内面を掘るのは限界がある
2. 代わりに**非言語の直感の痕跡**（なんとなく保存した画像、よく聴く音楽、視聴履歴）をAIに渡して分析させる
3. **AIに答えを聞くのではなく、答えが既に現れている素材を渡して整理・言語化させる**。この順番が肝
4. この手法の本質は自己分析ではなく**「なんとなく」を言葉に変える技術**。自己理解にもクリエイティブ発注にも応用できる

### 手順（4ステップ）

- **STEP1 — OS入力**: 生年月日、算命学・四柱推命、MBTI、簡単な経歴を入力。**肩書きは盛らない**（「上場企業の社長」ではなく「会社経営」。AIがイメージに引っ張られるため）。役割設定は「カウンセラー・編集者・音楽プロデューサー・ブランド戦略家・占術家の視点を統合した、私専用の直感分析エージェント」。断定ではなく「この素材を統合するとこういう無意識のテーマが浮かぶ」という形式で出力させる
- **STEP2 — 画像分析**: Pinterest等に「なんとなく」保存した画像群をまとめて渡す。1枚ずつではなく**画像群全体に共通する無意識のパターン**を読ませる。観点: 色味／光の質／時間帯／季節感／空間の広さ／都市と自然のバランス／人物との距離感／余白／懐かしさと未来感の比率／退廃と清潔の比率／静けさと祝祭の比率／孤独と繋がりの比率／回復したいのか・戦いたいのか・逃げたいのか・変わりたいのか。分析用に画像を選ばないこと（「なんとなく好き」ほど本音が出る）
- **STEP3 — 音楽分析**: プレイリストを「好み」ではなく**今の精神状態**として読ませる。感情のBPM、回復/戦闘/逃避/祝祭モードのどれが強いか、避けている感情、今聴くべき音楽・あえて聴かない方がいい音楽、処方プレイリスト
- **STEP4 — 統合診断**: OS・画像・音楽を統合し、深層心理／避けている課題／次に向かう方向／1週間の行動プラン／自分診断カルテ／内面を表す画像生成プロンプトまで出力させる

### 実例（記事内）

- ゆとりくん: ダークファンタジー的な画像群から「自然光がほぼ存在しない → 太陽の下に立つことの疲労」「作り込まれた閉鎖空間への逃避願望」と指摘され、経営者として矢面に立つ疲労をAIが名指しした。音楽のBPMからは「立ち止まって悲しみを感じることを避けている」
- 寺田氏: 診断結果は「着工が遅れる建築家」。企画・戦略・世界観は好きだが、投稿・リリースなど実行が遅い、という図星

### ECクリエイティブへの応用（寺田氏の転用）

- 「かっこいい感じにして」ではデザイナーに伝わらない → 修正10回超、制作費5万円のドブ捨ての実体験
- 参考画像群をAIに渡し「ブランド戦略家兼アートディレクター」として、共通する色味・光の質・時間帯と季節感・空間の広さ・コアインサイト（なぜターゲットがこれに惹かれるのか）を分析させ、**カメラマン/デザイナーがそのまま使えるディレクション指示書**として言語化させる
- 効果: 「ピンク」→「色褪せたヴィンテージフィルター越しのダスティピンク」、「明るく」→「自然光がほぼ存在しない、全てが発光する演出された光」。指示ではなく世界観そのものをチームで共有できる
- 実例: 寺田氏のムードボード分析 →「成功を誇示するラグジュアリーではなく、"余白のある成功者"の世界観。持っていることではなく、どう生きているかを感じさせるビジュアル」

### 教訓

- 論理的な100点の正論はすぐ真似され飽きられる。価値があるのは「なんとなく惹かれる」直感で、**AIはその翻訳機**
- 本当の効率化は、まず自分の内面（直感）を言語化することから始まる
- **言語化して満足するな。試すまでがセット**（「着工しない建築家」になるな。設計図は美しいが、宮殿は建て始めた日から美しい）

### この会社（副業コンテンツ事業）への応用アイデア

- **SNS/note/YouTubeの世界観設計**: 自分が保存している画像・よく見る動画をAIに分析させ、発信の「トーン&マナー指示書」を作る。投稿ごとにブレない世界観の言語化に使える
- **コンテンツネタ**: この「直感カルテ」体験自体が記事・投稿ネタになる（実践レポート形式は寺田氏の記事構成がそのまま参考になる: 権威の意外な使い方 → 手順公開 → 自分でやってみた → 失敗込みのオチ → 読者への問いかけ）
- **サムネイル・アイキャッチ発注**: ECクリエイティブ言語化プロンプトの手法は、YouTubeサムネやnoteのアイキャッチをデザイナー/AIに依頼する際の精度向上にそのまま転用できる

## 川邊健太郎氏に学ぶ「AI×ソロプレナー（1人企業）」（15:54 追記）

**出典**: 川邊健太郎氏（元ヤフー社長CEO・元ZHD社長CEO・元LINEヤフー会長、現LINEヤフー顧問。1995年21歳で「電脳隊」起業、孫正義氏の下で14年事業開発）がダイヤモンド・オンラインのYouTubeで語った「AI時代の1人起業」の解説記事。川邊氏は2026年6月の会長退任直後からAI×ソロプレナーを自ら実践中。

### なぜ「AI＋1人起業」なのか

- 出発点は組織の巨大化への問題意識: 人が増えるほどコミュニケーションコストが増え、責任が重くなりリスクを取りづらくなる。経営者は自分でサービスを作れなくなる
- AI以前は1人になっても自由に作れなかった（資金調達→エンジニア採用が必要）。**その工程が不要になったことが画期的な変化**

### 「業界×AI」より「AI×業界」が強い（掛け算の順番＝主役はどっちか）

- **業界×AI**: 既存業界のやり方が土台で、AIを道具として後付け。設計思想は昔のまま
- **AI×業界**: AIが動くことを前提に会社をゼロから設計。AIがエンジン（OS）、業界は扱う対象
- 例: 作業療法士がAIで管理システムを作るのは「業界×AI」で入り口にすぎない。AIが業務そのものを覚えると、最初からAI駆動前提で組むプレイヤーが現れ、**生産性の桁が変わる**ので後者が追い抜く
- 元ネタは「業界×ネット」より「ネット×業界」が勝った歴史。同じことがAIでも起きる
- **留保**: 規制で守られた業界（ライドシェア、弁護士相談など「人間しかやってはいけない」領域）は別。自由競争の領域に限った話

### 組織のつくり方: AIエージェントを「部下」にする

- 川邊氏は自分の会社でAIエージェントを部下として配置。構成は「すごくオーソドックス」で人間の組織図とほぼ同じ: **経理担当／法務担当／プロダクトマネージャー役／マーケティングリサーチ担当**
- PM役の下にさらに約10のエージェントをぶら下げ、POC（試作・検証）をどんどん回す体制
- AI駆動サービスは運営自体にトークン代がかかるため、**「今月どこまでトークンを使えるか」を経理エージェントに相談し、「これ以上は追加できません」と返される**——人間の会社と同じ予算のやり取りがエージェント間で生まれる

### 人間に残る3つの役割

1. **発信**: 自分がやると最もレバレッジが効く領域
2. **最終検証**: 例）ハンバーガー屋のレシピをAIに全部考えさせ、味見だけ人間がやる。味覚はAIに代替できない
3. **細かい手作業**: 設定画面の操作など
- 川邊氏は「役割が残っていること自体に安心感を覚える」と

### 一番効くのは資金より「意志の力」

- AIで最も重要なのは意志の力。「何を作りたいか」「どういう未来を実現したいか」が起点。トークン代はかかるが、意志さえあれば節約策やローカルLLMなどやりようはいくらでもある
- 孫正義氏のエピソード: 「金がかかるからそこまでやる金はない」と言うと**「俺たちが金を持っていたことが一度でもあるか。金はどこから持ってくればいいんだ」**と返された
- 「格差が広がる」と言いすぎるな——「どうせ自分にはできない」という抑制効果を生むから

### 出口とフィールドの選び方

1. **出口（イグジット）の前提が変わる**: IPOはネット産業に立脚したモデル。AI駆動型企業は挙動が違うのでIPOのあり方も変わる。どう変わるかはやりながら探す段階
2. **規制のない領域を選ぶ**: 日本ではライドシェアが象徴のように、規制領域はほぼ動かない。川邊氏は「日本のAI化はそこそこ止まり」とすら見る。だから規制のないフロンティア（ECなど）を選ぶ。**AI時代は"無人の荒野"がもう一度広がる**
3. **ボーングローバルで戦う**: 日本は制度改革が遅い。これからの1人起業は最初からグローバル前提が むしろチャンス

### この会社（副業コンテンツ事業）への応用アイデア

- **この `.company` はまさに「AI×業界」の実践**: 秘書・SNS・note・YouTubeの各部門をAIエージェントとして配置する構造は、川邊氏の「AIを部下にしたオーソドックスな組織図」と同型。既存のやり方をAIで速くする（業界×AI）のではなく、AIが動く前提で運営を設計する方向を意識する
- **人間（オーナー）が集中すべきは3役割**: ①発信（本人の顔・声で出す部分が最大のレバレッジ）②最終検証（投稿・記事の最終チェックと「味見」）③細かい手作業（各プラットフォームでの投稿操作など）。それ以外はAI側に寄せる設計にする
- **コンテンツネタとして強い**: 「元ヤフー社長が退任後に選んだのは1人起業」「AIエージェントの経理に予算を断られる」などフック力のある素材。直感カルテ（同日の学び①）と合わせて「AI活用の設計思想」シリーズが組める
- **意志の力の話は発信の軸になる**: ツールは誰でも使える時代、差を生むのは「何を実現したいか」という意志と実行力。前記事の「着工しない建築家になるな」と同じ結論に収斂しており、2本を貫くテーマとして使える

## カーパシー版Claude活用 — Software 3.0と「AIを働かせる設計思想」（16:05 追記）

**出典**: 「Vibe CodingからSoftware 3.0へ。Claudeを"英語で動く開発・研究・実行チーム"に変える方法」。アンドレイ・カーパシー（OpenAI創業メンバー、元Tesla AIディレクター）の講演・メモとClaude最新機能を接続した実践フレームの解説記事。筆者自身が「公式モードではなく再構成した実践フレーム」と明記している。

### 中心思想: Software 3.0

- **Software 1.0** = 人間が書くコード、**2.0** = ニューラルネットの重み、**3.0** = LLMへのプロンプト。自然言語が新しいプログラミング言語になる
- Claudeは「チャット相手」ではなく**自然言語で動く新しい実行環境**。「AIに聞く」ではなく「AIで作る」が起点
- カーパシー自身のワークフローも「80%手書き」から「80%エージェント」へ転換（2025年末頃が閾値・相転移だったという見方）

### 核心原則（7原則の要約）

1. **Claudeを新しいコンピュータとして扱う** — 質問ではなく、作らせる・動かせる成果物を出させる
2. **命令ではなく成功条件を与える** — 「これをやって」ではなく「何を満たせば成功か」。AIエージェントは明確なゴールに向かってループするのが得意
3. **テストを先に置く** — テストは自律的に動かすためのレール。通るまで反復させる
4. **人間は書く人から読む人へ** — 生成はAI、識別（審美眼・設計判断・セキュリティ判断）は人間。これを放棄するとAI生成物はすぐ負債化（"slopacolypse"）
5. **過剰な抽象化を必ず削る** — 実装後に「もっと単純にできないか」。1000行を100行に戻す勇気
6. **環境設計で決まる** — CLAUDE.md（プロジェクト憲法）、Skills（業務マニュアル）、Subagents（観点の分離: 実装するAIと疑うAIを分ける）、MCP（現実の業務システムへの感覚器官）
7. **時短ではなく行動範囲の拡張** — 本当の価値は「以前なら面倒でやらなかったこと」をやれるようになること（※川邊氏の学びとも共鳴）

### 実務で使える型

- **頼み方の進化**: 「バグ直して」→「目的+成功条件+進め方（探索→計画→テスト先行→実装→実行→報告）」の構造で渡す
- **XMLタグで文脈を構造化**: `<task>` `<context>` `<success_criteria>` `<constraints>` `<output>` に分けて渡す。プロンプトはSoftware 3.0のコードスタイル
- **単純化レビューを必ず挟む**: 作らせた後に削らせる、実装させた後に疑わせる、説明させた後に人間が読む
- **完了報告を信じすぎない**: 「完了と判断した根拠(実行コマンド・通ったテスト・確認ファイル・未確認リスク)を分けて報告して」と証拠を要求する
- **Vibe Codingの誤解を正す**: 「コードを読まなくていい」ではなく逆。大きく書かせるほど厳しく読む必要がある
- **回答ではなくArtifact**: 説明ではなく、ダッシュボード・プロトタイプ・ドキュメントなど「使える成果物」を残させる

### 非エンジニアへの応用（この記事の重要ポイント）

原則はコーディング限定ではない。営業（商談戦略）、マーケ（LP前の顧客理解）、人事（面接設計）、経営（月次資料の意思決定用整理）でも「成功条件を与え、構造化された成果物を出させ、レビューさせる」型がそのまま使える。**Software 3.0の民主化**。

### この会社（副業コンテンツ事業）への応用アイデア

- **秘書への依頼も「成功条件」型にする**: 投稿作成を頼むときは「バズる投稿を書いて」ではなく、目的・成功条件（例: 冒頭1行で悩みを提示／140字以内／CTAあり）・制約を渡す。逆に部門側のCLAUDE.mdに成功条件のデフォルトを書き込んでおけば毎回書かなくてよい
- **各部門CLAUDE.mdを「憲法」として強化**: 現在のCLAUDE.mdは運用ルール中心。禁止事項（誇大表現NG等）、品質基準、レビュー観点を追記すると出力が安定する
- **「作らせた後に削らせる」をコンテンツにも適用**: 投稿・記事の初稿後に「単純化レビュー」（冗長な段落、いらない装飾、主張のダブり削り）を必ず挟むワークフローにする
- **3記事の統合テーマが完成**: ①直感カルテ=素材を渡して言語化させる ②川邊氏=AI前提で組織を設計する ③カーパシー=成功条件を与えて働かせ、人間は読む・削る・判断する。「AIに聞く時代の終わり — 渡す・設計する・検証する」として発信シリーズが組める
